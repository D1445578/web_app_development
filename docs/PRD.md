# 產品需求文件 (PRD) - 校園遺失物查詢系統

## 1. 專案概述
- **背景與動機**：校園內經常發生學生遺失物品的狀況，拾獲者往往不知如何尋找失主或只能交給生輔組，缺乏一個即時、透明且一目了然的資訊共享平台。
- **目標用戶**：常把東西用不見的人、不小心遺失物品的師生、在校園內撿到物品的熱心人士。
- **核心價值主張**：提供一個可以快速登記遺失物與撿到物品，並進行自動配對與聯絡的校園專屬平台，大幅提高失物尋回效率。

## 2. 功能需求
以下為系統主要功能與使用者故事：

1. **遺失物登記**
   - 作為 **遺失物品的人**，我希望 **能夠快速登記遺失物的特徵、時間、地點並上傳圖片**，以便 **讓全校師生知道我正在尋找該物品**。
2. **撿到物品的登記**
   - 作為 **拾獲物品的人**，我希望 **能夠登錄撿到物品的資訊與拾獲地點**，以便 **讓失主可以確認並與我聯繫**。
3. **列表瀏覽**
   - 作為 **系統使用者**，我希望 **能夠瀏覽目前所有的遺失與拾獲物品列表**，以便 **隨時關注平台上是否有我遺失的物品**。
4. **尋找功能 (搜尋與篩選)**
   - 作為 **尋找物品的人**，我希望 **能夠透過輸入關鍵字及篩選地點**，以便 **快速在大量紀錄中找到可能符合的物品**。
5. **配對提示 (自動比對)**
   - 作為 **建立登記的人**，我希望 **系統能自動比對資料（如類型、地點）並提示可能有相符的紀錄**，以便 **節省我手動尋找的時間**。
6. **聯絡功能**
   - 作為 **確認物品相符的人**，我希望 **能夠看到對方提供的聯絡方式（如 IG、社群帳號）**，以便 **私下聯繫面交與歸還事宜**。

## 3. 非功能需求
- **技術限制**：
  - 後端：須使用 Flask 框架。
  - 前端渲染：使用 Jinja2 模板引擎與純 CSS/JS (或搭配簡單的 UI 框架如 Bootstrap/Tailwind)。
  - 資料庫：SQLite (適合初期輕量型應用儲存)。
- **效能考量**：搜尋與列表讀取需快速（反應時間 < 2秒）；上傳的圖片需限制大小（例如不超過 2MB 或 5MB）以防伺服器超載。
- **安全考量**：防範基本的 SQL Injection 和 XSS 攻擊；保護聯絡資訊，可設定隱私選項。

## 4. MVP 範圍 (Minimum Viable Product)
- **Must Have (必須具備 - 核心功能)**
  - 遺失物與拾獲物登記表單（含圖片與基本選項）。
  - 所有圖文列表瀏覽。
  - 顯示聯絡功能（使用者輸入 IG 帳號或 Email）。
- **Should Have (應該具備 - 增強體驗)**
  - 尋找與篩選功能（關鍵字、地點、時間區段）。
- **Nice to Have (有會更好 - 進階功能)**
  - 配對提示自動比對機制（基於關鍵字與標籤）。
  - 系統內建雙方留言板或推播通知。
